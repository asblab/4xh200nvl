Scope

This page documents the measurement process behind the results: the hardware and software configuration, the exact benchmark invocations, the complete result tables, and the supporting analyses. All measurements were taken 2026-07-17 through 2026-07-21 on a single 4x H200 NVL node. Measurement closed 2026-07-21 on a frozen stack: vLLM v0.23.0, GPU driver 595.71.05, CUDA 13.2, and a single pinned version of the agent harness. The numbers are free to use.

Test environment

Throughput methodology

All GPU throughput rows were produced by vLLM's bench client running against a live server on the same node:

vllm bench serve --backend openai-chat --dataset-name random \
  --random-input-len 512 --random-output-len 256 \
  --ignore-eos --seed 42

Two load points per configuration: single-stream is concurrency 1 over 3 prompts; c8 is concurrency 8 over 24 prompts. All servers were launched with --gpu-memory-utilization 0.90 --max-model-len 32768. TTFT is time to first token.

Multi-GPU rows use tensor parallelism: each layer's weight matrices are sharded across the GPUs, every GPU computes its shard for every token, and the partial results are combined with an all-reduce after each layer. All 4-GPU launches additionally required --disable-custom-all-reduce on this platform (see the initialization-stall section below).

Model-specific serving notes

DeepSeek-V4-Flash: 159 GB of bf16 weights; --kv-cache-dtype fp8 is required by its attention implementation in this vLLM build, so its rows are bf16 weights with an fp8 KV cache. The speculative-decoding row enables the model's own draft head (one speculative token): +9% single-stream, +21% at c8, tool calls unaffected. Qwen3-Coder-480B-A35B-Instruct: 482 GB of FP8 weights, 4 GPUs. GLM-5.2: 387 GB, int4 weights with bf16 activations (compressed-tensors pack-quantized), 4 GPUs; the reasoning output requires a think-tag reasoning parser or the reasoning text leaks into message content. gemma-4-31B is dense (not a mixture-of-experts), which is why its throughput sits far below the similarly sized MoE models. The CPU row is llama.cpp with a Q8_0 quantization, 128 threads, --numa distribute, zen4 backend.

Concurrency ladder

Concurrent-stream capacity

Capacity per configuration is the largest concurrency at which per-stream output stays at or above 20 tok/s. Two request shapes were measured: 512-token prompts with 256-token outputs (light load) and 8,192-token prompts with 512-token outputs (the shape of real coding-agent requests). Multi-GPU rows run separate single-model instances simultaneously, one bench client per instance, and sum the per-instance knees; tensor parallelism is not used for capacity scaling. Measurement rules that mattered: every invocation uses a fresh random seed (see the prefix-cache note below); only the row’s instances run during a measurement (an idle engine on another GPU reduced active-instance throughput by about 10%); knee points near the threshold were confirmed 2-of-3.

Prefix-cache inflation. vLLM enables automatic prefix caching by default, and the bench client generates its prompt set from a seed. Re-running a bench with the same seed therefore lets the server skip prefill for prompts it has seen, and the effect scales with prefill cost: for DeepSeek-V4-Flash (3.3 s prefill at 512 tokens) repeated-seed runs measured 2–3× higher throughput than fresh-prompt runs at the same concurrency, and its capacity figures were initially overstated 2× (192 versus a true 96 at 512-token prompts). The discrepancy was caught when two identical instances on different GPU pairs disagreed 3× and both hardware and interconnect checks came back clean; a fresh-seed run on the “fast” instance reproduced the “slow” one exactly. Every capacity number in the tables above was measured with unique prompts. Real agent traffic re-sends its own growing conversation prefix and does benefit from the cache, so production capacity sits between these cold floors and the cached ceilings.

Multi-GPU decode analysis

Qwen3-Coder-30B-A3B activates ~3.3B parameters per token. In bf16 (2 bytes per parameter), reading the active weights once per token moves ~6.6 GB per token (est.). At the measured single-GPU, single-stream rate of 212.4 tok/s, that is ~1.4 TB/s of effective weight-read bandwidth (est.), about 29% (est.) of the H200's ~4.8 TB/s specified HBM bandwidth. The estimate counts active-weight reads only; KV-cache and activation traffic are excluded, so it is a lower bound on total memory traffic.

If single-stream decode were bandwidth-bound, sharding the weights across N GPUs would cut per-GPU weight traffic by N and permit up to an N-fold speedup. Measured single-stream scaling (computed from the throughput table): Qwen3-30B +5.7% at 2 GPUs and +12.4% at 4 GPUs over 1 GPU; gemma-4 +4.9% and +13.5%. Tensor parallelism inserts an all-reduce after each layer on every token; the measured deltas are consistent with per-token latency terms, not HBM bandwidth, limiting single-stream decode at this model size.

At concurrency 8 the picture differs by model: Qwen3-30B rises from 870.0 (1 GPU) to 1001.3 (2 GPUs, +15.1%) and then drops slightly to 987.9 at 4 GPUs (below the 2-GPU value); gemma-4 continues scaling, 998.7 to 1224.5 (+22.6%) to 1477.3 (+47.9% over 1 GPU).

4-GPU initialization stall

Symptom. 4-GPU tensor-parallel launches stalled during vLLM engine initialization and never reached a serving state. The same model and flags on 1 and 2 GPUs initialized normally.

Isolation. The stall was narrowed to vLLM's custom all-reduce path during startup: disabling only that component, with everything else unchanged, allowed initialization to complete.

Resolution. All 4-GPU launches on this platform run with --disable-custom-all-reduce, which falls back to NCCL for cross-GPU reductions. Every 4-GPU row in the tables above was collected under this flag.

GPU-wedge observation. Killing a stalled launch did not always return the GPUs to a clean state; a wedged GPU blocked subsequent launches on that device until the stale state cleared. Stalled 4-GPU launches were therefore treated as a state to avoid rather than retry.

Second occurrence. After an unrelated node reboot, the same stall class appeared at just 2 ranks: a 2-GPU launch hung in NCCL initialization for 20+ minutes, and after a graceful stop the driver reported unreadable utilization on the two affected GPUs, with a diagnostic container frozen in kernel state and unkillable from userland. A further reboot (with a driver update to 595.71.05) cleared it, and a 4-rank allreduce smoke test passed in seconds afterward. The operating rule that came out of both incidents: after any reboot, prove multi-rank NCCL with the 2-minute smoke test before the first multi-GPU model launch, and never hard-kill a process that may be inside a collective.

Tool calling and the streaming-parser failure

Each model was probed at temperature 0 through vLLM's tool-call parsers before any agentic run.

The Qwen3-Coder-30B case is a streaming-specific parser failure. With the qwen3_coder parser, 9 of 30 agentic tasks ended at turn 1: the model's first response contained a call the streaming parse path dropped because the opening tool_call tag was missing from the emitted text. The 10 failing transcripts, re-fed through the non-streaming path of both qwen3 parsers, parsed 10 for 10 — the model's output was recoverable; the incremental streaming parser was the failure point.

Switching the same model to the qwen3_xml parser and rerunning the full suite produced 0/30 turn-1 failures. The pass rate did not change (4/30 under both parsers), and median turns per session rose from 6.5 to 9: sessions that previously ended at the first dropped call instead ran to completion.

Reliability probes. Each of the six models also ran a 50-prompt tool-call probe at temperature 0: 38 well-formed prompts across three tool schemas, plus 12 prompts that deliberately mismatch tool and target (a directory passed to a file reader, and similar). All six models produced correctly structured calls on all 38 well-formed prompts — 228 of 228 with zero malformed calls. On the mismatched prompts the models differ in behavior, not correctness: Qwen3-Coder-480B refused 9 of 12 with an explanation, GLM-5.2 and Qwen3-30B refused 2, DeepSeek-V4-Flash 1, and both gemmas called the tool as asked every time.

CPU serving

The CPU baseline serves the same Qwen3-Coder-30B model through llama.cpp b10064: Q8_0 quantization, 128 threads across both EPYC 9555 sockets, --numa distribute, zen4 backend. llama-bench measures pp512 at 196.9 tok/s and tg128 at 39.6 tok/s. Under the same serving bench as the GPU rows, single-stream decode is 26.8 tok/s (TTFT 1868 ms) and c8 aggregate is 94.3 tok/s (TTFT 4912 ms).

Bandwidth arithmetic: at Q8_0, roughly 1 byte per parameter (est.), the ~3.3B active parameters imply ~3.3 GB of weight reads per token (est.). The llama-bench decode rate of 39.6 tok/s corresponds to ~130 GB/s of sustained weight reads (est.); the served single-stream rate of 26.8 tok/s corresponds to ~88 GB/s (est.). Against the single-GPU bf16 rows, the measured deltas are 212.4 vs 26.8 tok/s single-stream (~7.9x) and 870.0 vs 94.3 tok/s at c8 (~9.2x).

Agentic evaluation methodology

The suite is 30 coding tasks derived from Exercism exercises: 14 Python, 8 JavaScript, 8 Go. A terminal coding agent runs each task at stock defaults with no extensions; the product name is withheld under this project's publication policy. Hidden tests execute after the agent finishes and alone decide pass or fail. Zero retries, server-default sampling, and a 600-second cap per task.

Null lane. To confirm the pass bit is sound, the hidden tests were run against the untouched task stubs: 35/35 fail. A passing task therefore requires the agent to have produced a working change.

Suite versions. Suite v1 rows are from 2026-07-17. Suite v2 contains identical tasks and adds node-to-bun PATH shims; later rows use v2.

Publication policy. Every completed run is published as a table row; there is no score threshold. Early rows in evals.csv include a second harness that was used briefly and then dropped; those rows are retained as data.

Evaluation results and session shape

DeepSeek-V4-Flash session shape. 18/30: 9/14 Python, 6/8 Go, 3/8 JavaScript. Median 13.5 turns per session; 49 errored tool calls out of 417, each recovered from; zero protocol failures.

GLM-5.2. The lowest tool-call error rate measured (3 errored calls in 163) and zero protocol failures. Sessions are slow — 127 s median wall clock, 8276 median output tokens — because the model emits reasoning tokens before acting.

Qwen3-Coder-480B. 0 protocol failures. In each run, 2–3 tasks failed on test-suite timeouts — submitted solutions that loop indefinitely. Sessions were short; the model typically wrote a solution once and did not iterate against test feedback.

gemma-4-31B. The most token-efficient sessions measured: 1829 median output tokens, about a fifth of the 26B’s 6831, at a higher pass rate (8/30 versus 5/30).

gemma-4-26B. Long sessions that continued through 450 errored tool calls out of 1041, recovering each time.

Thermals and power

GPU temperature, power, and clock samples were logged every 10 seconds through the heaviest measurement windows: 4,840 samples over 4.2 hours, including the peak-load capacity tests. Maximum observed temperature was 70°C (three samples; p99 62–64°C per GPU) and maximum board power 590 W against the 600 W limit. No thermal throttling was observed; the only clock reduction was standard power-limit governance to 1425 MHz on 8 busy samples. At the peak measured load (51,400 aggregate output tok/s across four instances, cache-hot prefills) the node produced 21.9 output tokens per second per GPU-watt, using the peak all-GPU power draw of 2,346 W; the fresh-prompt peak of 37,700 tok/s was measured after power logging stopped, so no per-watt figure is quoted for it.

Operational incidents

Failure modes encountered while running the lab, and what fixed each. All were recoverable; none damaged data that had been committed.

Local scratch exhaustion, five times. The node’s local temporary space is 2 GB, and three separate growth mechanisms filled it during evaluation runs: per-session agent databases (~10 MB per task), task workspaces accumulating JavaScript dependency trees across runs, and a runtime that extracts a ~5.6 MB native library per process and only removes it on graceful exit — every timeout-killed process leaked one. The durable fix was per-task cleanup of all three classes inside the runner; a sweep of leaked libraries recovered 90% of the volume at its worst.

Shared-filer write protection, twice. The NFS model store sat near 98% full for most of the project, and its cluster twice switched the export to read-only under a capacity-protection state — refusing all writes including deletes, so the condition could not be relieved from the client side. One flip killed an evaluation run mid-write, and its traceback was unwritable because the log lived on the same filesystem. Both cleared without client action; the durable fix was the export being grown 12%, moving it well below the protection watermark. On shared storage, capacity headroom is availability.

Escaped test processes. Submitted solutions that loop forever are killed by the harness’s test timeout — but the timeout killed only the direct child, and the grandchild running the solution survived: three orphans were found consuming a full CPU core each for 30+ hours on a shared node. Fix: run tests in their own process group and kill the group.

Measurement traps. Two are documented in their own sections: the streaming tool-call parser that silently dropped model output (tool calling, above), and prefix caching inflating repeated-prompt benchmarks by up to 3× (capacity, above). Both were caught by cross-checking two independent measurements that should have agreed and did not.
