Summary

Six open-weight models, three axes: end-to-end capability as a coding agent, tool-call reliability, and concurrent capacity. Capacity is counted in concurrent streams: simultaneous requests each held at 20 output tokens/s or better, measured with unique prompts on every run so that every prefill is real work (repeated prompts let the server’s prefix cache skip prefills and inflate results — details in the methods). The primary numbers use 8,192-token prompts with 512-token outputs, the shape of real coding-agent requests; the 512-token columns are the light-load ceiling. Multi-GPU counts come from separate single-model instances benched simultaneously, not from tensor-parallel scaling.

Prompt size dominates capacity: at agent-sized prompts every model supports 4–28× fewer streams than at short prompts, because prefill and KV-cache space, not decode, become the limit. The two MoE coding models hold up best; the dense 31B and the two 4-GPU models drop to single digits.

Summary of findings:

  • DeepSeek-V4-Flash produced the strongest agent results: 15–18 of 30 suite tasks across four runs (mean 16) against 5–11 for every other model, with 24 concurrent agent-sized streams on 2 of the 4 GPUs (96 at light load).
  • GLM-5.2 was the second-strongest agent and had the lowest tool-call error rate measured (3 errored calls in 163) and a 38 ms first token at 387 GB. Reasoning tokens make each completed task roughly 2.5× slower than DeepSeek-V4-Flash.
  • The 30B-class models are fast but score low as agents. One GPU sustains 64 agent-sized streams of Qwen3-Coder-30B (448 at light load); its pass rate is 5 of 30 under every environment, parser, and precision tested.
  • The hardware was not a limiting factor. Peak observed: 37,700 output tokens/s aggregate with fresh prompts (51,400 with cached prefixes), no thermal throttling over 4+ hours of sustained load (max 70°C, 600 W limit).

Agent capability

The suite is 30 Exercism-derived tasks (14 Python, 8 JavaScript, 8 Go) driven end-to-end by a terminal coding agent at stock defaults: no extensions, server-default sampling, zero retries, 600-second cap per task, hidden tests applied after the agent finishes. As a control, the hidden tests fail 35/35 against untouched stubs. Every completed run is a row; there is no publication bar.

Two details from inside the runs. gemma-4-31B used the fewest output tokens per task (1,829 median, versus 6,831 for the 26B) and was the strongest of the sub-100 GB models. The FP8 row measures the quantization cost directly: the score is identical to bf16, and throughput moves −8% single-stream / +13% batched.

Tool calling

A coding agent works by issuing structured tool calls; a model that cannot produce them cannot act as an agent regardless of code quality. Each model was probed at temperature 0 through vLLM’s tool-call parsers before any suite run.

Serving speed

Speculative decoding. DeepSeek-V4-Flash carries its own draft head. Enabling it measured +9% single-stream and +21% at 8 streams, with tool calls unaffected.

Long context. DeepSeek-V4-Flash decode speed at 16k / 64k / 128k / 240k tokens of prompt fill: 81.9 / 49.6 / 35.4 / 20.2 tok/s, with the first token in 1.2 / 3.7 / 5.6 / 11.1 seconds. Retrieval checks passed 5/5 at 32k and 128k of fill and 4/5 at 250k. For the 30B-class models, growing the prompt from 1k to 28k tokens costs 39–56% of decode speed.

Interference between instances. Two single-GPU replicas cost each other little until the host saturates; two 2-GPU DeepSeek-V4-Flash instances showed no measurable mutual interference (their traffic stays inside NVLink pairs). An idle vLLM engine on another GPU reduced active-instance throughput by about 10%.

Thermals. 4,800+ samples across the heaviest hours: max 70°C, p99 ≤64°C, peak 590 W against the 600 W limit, no thermal throttling observed; the only clock reduction was standard power-limit governance. At the measured peak load the node produced 21.9 output tokens per second per GPU-watt.
